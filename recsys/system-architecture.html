Recall 召回层

召回层: 从海量到候选

召回层的目标是从百万级候选池中快速筛选出千级相关候选，核心指标是覆盖率 (Recall@K) 而非精度。要求延迟极低 (<10ms)，因此不能用复杂模型，通常采用向量检索 + 多路召回。

双塔模型 (Two-Tower)

用户塔和物品塔分别编码为向量，离线计算物品向量建索引，在线只需计算用户向量 + 向量检索。这是实现亚毫秒延迟的关键。

$$\text{score}(u, i) = \langle f_{\text{user}}(\mathbf{x}_u), f_{\text{item}}(\mathbf{x}_i) \rangle$$

两个塔必须完全独立，不能有交叉特征 — 否则无法预计算物品向量

Ranking 排序层

粗排 + 精排: 模型的主战场

排序层是全部 CTR/CVR 模型的部署位置。粗排用轻量模型快速筛选，精排用全量模型精细打分。我们学习的 FM/DeepFM/DIN/DCN/MMOE/PLE/PEPNet 全部用在精排层。
粗排 (Pre-Ranking)
精排 (Fine-Ranking)

粗排的设计约束

  • 候选量 ~1000，延迟 < 15ms
  • 模型必须轻量：双塔、浅层 DNN、知识蒸馏
  • 不用复杂的注意力机制 (太慢)
  • 目标：保留精排会选出的 Top 候选

常用方法

知识蒸馏：用精排模型的预测作为 soft label 训练轻量学生模型
特征裁剪：只用 Top-K 重要特征
模型简化：浅层 MLP、无序列特征

精排: 模型全家桶

精排层是模型能力的主战场。候选量 ~200，可以用 GPU 推理，模型复杂度不受限。

Re-Ranking 重排层

重排层: 从排序到展示

精排输出的 Top-20 候选还需要经过重排才能最终展示。重排层不再优化单个广告的 CTR，而是优化整页/整个列表的体验和收入。

多样性

避免连续展示同一类别/广告主的广告。MMR (Maximal Marginal Relevance)、DPP (Determinantal Point Process)

频次控制

同一广告对同一用户的展示频次上限。过度曝光导致用户疲劳和负体验。

竞价与定价

GSP (广义二价) / VCG 机制。确保广告主激励兼容，平台收入最大化。

校准 (Calibration)

重排层还负责最终的预估值校准。精排输出的 pCTR/pCVR 可能存在系统性偏差，需要通过 Platt Scaling 或 Isotonic Regression 校准后才能用于出价和收费。

$$\text{eCPM} = \text{pCTR}_{\text{calibrated}} \times \text{pCVR}_{\text{calibrated}} \times \text{target\_CPA}$$

PCOC (Predicted/Observed CVR ratio) 应接近 1.0，否则广告主 ROI 失控

Feature Engineering 特征工程

特征体系

特征工程是排序模型效果的基石。好的特征往往比更复杂的模型架构带来更大的提升。广告系统的特征通常分为四大类。
类别来源示例更新频率
用户画像 注册信息 + 行为聚合 年龄、性别、城市、兴趣标签、消费力 天级
用户行为 实时行为流 最近 50 次点击/购买序列、Session 内行为 实时 (Flink)
物品/广告特征 广告主上传 + 统计 类目、价格、品牌、历史 CTR/CVR、素材质量分 小时级
上下文特征 请求时生成 时间、地点、设备、网络、App 版本、页面位置 请求级

特征处理方式

类别特征 (Categorical)

通过 Embedding 层映射为稠密向量。
Embedding 维度通常 8-64，取决于类别基数。
高基数特征 (如 user_id) 需要 hash 或频率截断。

连续特征 (Continuous)

标准化/分桶/对数变换。
有些系统将连续特征也离散化后走 Embedding。
常用分桶方式：等频、等距、自适应 (树模型分桶)。

特征服务架构

特征更新链路:

离线特征 (天级):  Spark → Hive → Redis Bulk Load
近线特征 (分钟级): Flink → Kafka → Redis Streaming
实时特征 (请求级): 请求参数直接解析 (时间/位置/设备)

Feature Store 设计:
  Key:   user_id / item_id / cross_key
  Value: Protobuf 序列化的特征向量
  TTL:   user=24h, item=1h, realtime=session
  SLA:   p99 < 3ms (Redis Cluster)
Monitoring 监控与校准

模型监控体系

模型上线后最重要的事情是持续监控。模型衰减 (Model Degradation) 是必然的——用户行为变化、新广告上线、季节性波动都会导致模型效果下降。

核心监控指标

指标含义报警阈值处理方式
AUC 排序能力 (越高越好) 日环比下降 > 0.5% 检查数据管道 → 重训模型
PCOC 预估/实际转化率比值 偏离 1.0 超过 5% 触发在线校准 (Platt Scaling)
GAUC 分用户 AUC 加权平均 日环比下降 > 0.3% 检查用户侧特征漂移
Log Loss 概率校准质量 上升 > 2% 全链路校准检查
特征覆盖率 特征缺失比例 某特征缺失 > 10% 检查数据源 / Feature Store
推理延迟 模型 p99 延迟 超过 SLA 的 80% 扩容 GPU / 启用降级

校准 (Calibration) 详解

Platt Scaling

学习两个参数 a, b：\(p_{cal} = \sigma(a \cdot logit + b)\)
在验证集上拟合。简单有效，适合全局校准。

Isotonic Regression

非参数校准，学习保序映射函数。适合模型输出分布不均匀的场景。无需假设函数形式。

Field-Aware Calibration

按特定维度 (广告主、类目、时段) 分别校准。例如：电商广告和游戏广告的 CVR 分布差异很大，全局校准不够，需要分域校准。
